# Analysis Skeleton · 分析推进骨架

The advance order for a completed course-assignment analysis. The post derives
from artifacts; it is not a second attempt at the assignment.
本文件给出已完成课程作业分析的推进顺序。帖子从工件推出；不是第二次代做作业。

---

## Stanford reference profile + portable fallback | Stanford 参照 profile＋可迁移回退

Activate the **Stanford reference profile** only when the live repository matches
the source workflow's analysis-artifact constellation; Stanford identity alone
is insufficient. A credible match links validated bilingual/overview guides under
`_2_Assignment Guides`, a completed submission artifact, and the live
`_3_Assignment analysis (课程详解CN帖子用)` convention or a completed A1 analysis
anchor. One institution name or path does not establish the profile. When matched,
derive the course code; read the bilingual problems, Overview, then completed
submission in that order; use the CN Guide, module summaries, and grader as
secondary evidence. Preserve thematic order, four section anchors, stepwise
Written derivations, Coding mechanism analysis, Unicode/plain-text mathematics,
ordinary Markdown, no hidden-test details, and no complete ethics answers.
只有真实仓库匹配源工作流的分析工件组合时，才启用 **Stanford 参照 profile**；仅有 Stanford
机构身份并不足够。可信匹配须把 `_2_Assignment Guides` 下已验证的双语题面/Overview、
已完成提交工件，同真实的 `_3_Assignment analysis (课程详解CN帖子用)` 约定或已完成 A1
分析锚点连接起来；单个机构名或路径不能确立 profile。匹配后，推导课程代码，并依次读取
双语题面、Overview、完成提交；CN Guide、Module Summaries 与 grader 作次级证据。保留
按主题组织、每节四锚、Written 逐步推导、Coding 机制分析、Unicode/纯文本数学、普通
Markdown、无 hidden-test 细节及不提供完整伦理答案等约束。

Every nonmatching repository — including another Stanford course — preserves the
same evidence-to-explanation chain, but discovers its prompt, rubric, submission
forms, mathematical convention, section anchors, and publication target locally.
任何不匹配的仓库——包括另一门 Stanford 课程——都保留同一证据—解释链，但从本地工件
发现题面、rubric、提交形式、数学约定、章节锚点与发布目标。

## Order of advance | 推进顺序（默认骨架）

1. **Pass the completion gate** — discover course/assignment identity, the
   authoritative prompt or validated guide, the grading structure, the actual
   completed submission artifacts, and the analysis destination. Submission may
   be code, notebook, written PDF/Markdown/DOCX, experiment outputs, presentation,
   or a mixed set. Read prompts/guide first, overview or rubric second, completed
   artifacts third; read course notes and grader feedback only as needed. If a
   relevant artifact is incomplete, do not write a definitive analysis of it.
   **通过完成闸门**——发现课程/作业身份、权威题面或已验证 Guide、计分结构、真实已完成
   提交工件与分析目标目录。提交可以是代码、Notebook、书面 PDF/Markdown/DOCX、实验输出、
   演示文稿或混合集合。先读题面/Guide，再读 Overview/rubric，最后读完成工件；课程笔记与
   grader 反馈按需读。相关工件未完成时，不写确定性分析。

2. **Close the coverage map** — inventory every smallest graded unit in the
   following ledger. The `Status` set is fixed: `complete`, `partial`, `excluded`,
   `missing`, `conflicted`.
   **闭合覆盖图**——用下列台账记录每个最小计分单元。`Status` 固定为：`complete`、
   `partial`、`excluded`、`missing`、`conflicted`。

   | Source unit ID · 来源单元 ID | Type + weight · 类型＋权重 | Concept · 概念 | Evidence address · 证据地址 | Analysis job · 分析任务 | Status · 状态 |
   |---|---|---|---|---|---|
   | canonical local ID · 本地规范 ID | source value · 来源值 | one mechanism · 一个机制 | file/page/section/line · 文件/页/节/行 | derive/explain/evaluate/bound · 推导/解释/评估/标界 | fixed set only · 仅用固定词面 |

   Code and derivation are only two possible types; also cover experiment, design,
   interpretation, proof, report, and presentation tasks. Personal reflection,
   ethics, peer feedback, or AI-tutor prompts receive design-intent and reasoning-
   frame analysis only unless the user explicitly supplies an answer to analyze.
   Hidden tests and private grader details yield no publishable content.
   代码和推导只是两种类型；实验、设计、解释、证明、报告与演示任务也须覆盖。个人反思、
   伦理、同伴反馈或 AI-tutor 题只分析设计意图与思考框架，除非用户明确提供答案供分析。
   hidden test 与私有 grader 细节不产生可发布内容。

3. **Derive the knowledge chain** — group graded units by conceptual dependency,
   not numeric order. Adapt a chain to the assignment, for example:
   **推出知识链**——按概念依赖而非题号分组。按作业适配链条，例如：

   `representation → objective → method → implementation/experiment → evidence → evaluation → boundary`
   `表示 → 目标 → 方法 → 实现/实验 → 证据 → 评估 → 边界`

   Each chapter inherits the prior result and creates the next question. If two
   chapters can swap without damage, the chain is not ready.
   每章继承上章结论并逼出下一问。若两章可无损互换，知识链尚未成立。

4. **Anchor and expose every section** — before the first subsection of each
   chapter, use a 3–5 sentence bridge: where the chapter sits, what ability it
   creates, and how it joins its neighbors. At each section entry, show the source
   task ID, score/weight, and type near the prerequisite and why-it-matters anchors.
   Then separate prompt from explanation visibly. Under the Stanford reference
   profile, use `**题目**` for the source task and `**解析**` for Written work or
   `**思路与代码**` for Coding work; mixed sections repeat the pair. Outside the
   profile, use repository-language equivalents but keep both roles bold and
   distinct. The prompt block contains no analysis. After evidence, close with the
   earned takeaway and next-use anchors. Labels may adapt; functions may not vanish.
   **锚定并显露每节**——每章首小节前用三至五句搭桥：本章位置、新增能力及与前后章的连接。
   每节入口在前置知识与为什么学两个锚附近，展示来源题号、分值/权重与类型；随后明确分隔题面
   与分析。Stanford 参照 profile 固定用 `**题目**` 承载来源任务，Written 用 `**解析**`、
   Coding 用 `**思路与代码**`；混合章节依次重复。profile 外可用仓库语言等价词，但两类角色
   必须加粗且分明。题面块不含分析。证据之后，以经证据赢得的本节小结与下一用途两个锚收束。
   标签可适配；功能不可消失。

5. **Make reasoning visible** — for derivations/proofs use:
   `question → motivation → premises → one licensed step at a time → result →
   intuition → failure/boundary → connection`. For empirical tasks use:
   `question → setup → measure → observation → interpretation → rival → limit`.
   State why the reasoning is needed before its first symbol, chart, or result.
   A final answer without its path is not analysis.
   **让推理可见**——推导/证明采用：`问题 → 动机 → 前提 → 每次一个有依据步骤 → 结果 →
   直觉 → 失败/边界 → 连接`；实证任务采用：`问题 → 设置 → 测量 → 观察 → 解释 → 竞争解释 →
   局限`。第一个符号、图表或结果前先说明为何需要该推理。只有答案没有路径，不是分析。

6. **Explain completed artifacts as mechanisms** — state each artifact's role,
   interface or rubric constraint, inputs/outputs, and strategy before excerpts.
   Quote only load-bearing code, prose, figures, tables, or slides. Map each to
   the formula, invariant, design constraint, causal claim, or rubric criterion
   it realizes. Discuss alternatives only when they expose a real tradeoff. Use
   ordinary file/page/section/line addresses, not editor-only syntax.
   **把完成工件写成机制**——片段前先写工件角色、接口或 rubric 限制、输入输出与策略。只摘
   承重的代码、正文、图、表或幻灯片。把每个片段映射到它实现的公式、不变量、设计约束、
   因果主张或 rubric 条目。只有真实权衡才讨论替代方案。用普通文件/页/节/行地址，不用编辑器
   专属语法。

7. **Earn cross-task insight** — compare at least two graded units before naming
   a shared structure. Use a table only for a real mapping such as theory ↔ artifact,
   assumption ↔ failure, method ↔ evidence, or rubric ↔ design. Then state
   Methodological Gains: prior assumption, artifact-supported revision,
   transferable lesson. First person is permitted only for experience supplied by
   the user or directly evidenced by revisions/artifacts. If fewer than two
   comparable units exist, mark Cross-Task Insights not applicable; invent none.
   **赢得跨题洞见**——至少比较两个计分单元后才命名共同结构。表格只用于理论↔工件、假设↔失败、
   方法↔证据或 rubric↔设计等真实映射。随后写方法论收获：先前假设、工件支持的修正、可迁移
   教训。第一人称仅用于用户提供或修订工件直接证明的经历。若不足两个可比单元，则标记
   跨题洞见不适用；不得虚构。

8. **Pass the portable release gate** — use the repository's publishable Markdown
   and mathematical convention; prefer Unicode/plain text when the destination
   lacks LaTeX, and preserve LaTeX only when the target renders it reliably.
   Include a symbol/term lookup whenever notation would otherwise be undefined,
   plus the knowledge map, cross-task insight when earned, methodology, and factual footer.
   Let length follow assignment scope and analytical depth. Scan for hidden tests,
   unsupported biography, missing graded units, orphan excerpts, broken links,
   and platform-specific syntax.
   **通过可移植发布闸门**——使用仓库可发布的 Markdown 与数学约定；目标不支持 LaTeX 时优先
   Unicode/纯文本，只有目标可靠渲染时才保留 LaTeX。记号若不列就会失去定义，则必须提供
   符号/术语速查；并包含知识地图、经证据赢得时的跨题洞见、方法论与事实 footer。篇幅服从
   作业范围与分析
   深度。扫描 hidden test、无据经历、缺失计分单元、孤儿片段、坏链与平台专属语法。

## Default document chain | 默认文档链

`Metadata → Contents → Symbol/Term Lookup (if needed) → Preface → Knowledge Map → Thematic Chapters → Cross-Task Insights (if earned) → Methodological Gains → Footer`
`元数据 → 目录 → 符号/术语速查（按需）→ 前言 → 知识地图 → 主题章节 → 跨题洞见（经证据赢得时）→ 方法论收获 → 尾注`

The preface states the assignment's real training objective and organizing
question. It does not invent a dramatic personal conversion.
前言说明作业真正训练的能力与组织问题。不虚构戏剧性的个人转变。

---

## Stop conditions | 停止条件

- Incomplete relevant artifacts yield a gap report or provisional outline, not
  fabricated analysis. | 相关工件未完成时只交缺口报告或暂定提纲，不编分析。
- If prompt, rubric, and submission disagree, expose the disagreement; do not
  silently normalize it. | 题面、rubric 与提交冲突时显式呈现，不静默归一。
- Open personal answers remain the student's own work; analyze the frame unless
  the user supplies content for review. | 开放个人答案属于学生本人；除非用户提供内容供审阅，
  否则只分析框架。

## Self-check | 自检清单

- [ ] Were authoritative prompt/rubric and completed artifacts read in order? |
      是否按顺序读取权威题面/rubric 与完成工件？
- [ ] Is every graded unit mapped, with exclusions named? |
      每个计分单元是否映射，排除项是否点名？
- [ ] Do chapters follow conceptual dependency rather than numbering? |
      章节是否服从概念依赖而非题号？
- [ ] Does every section carry the four functions and visible motivation? |
      每节是否有四项功能锚与可见动机？
- [ ] Does every result show its derivation, experiment, or evidence path? |
      每个结果是否展示其推导、实验或证据路径？
- [ ] Does every excerpt map to theory, invariant, constraint, claim, or rubric? |
      每个片段是否映射到理论、不变量、约束、主张或 rubric？
- [ ] When present, are cross-insights supported by multiple tasks; otherwise is
      not-applicable explicit, with no invented experience? | 跨题洞见存在时是否由多任务支撑；
      不适用时是否明确标记，且无虚构经历？
- [ ] Is the output portable, complete, and free of private-test leakage? |
      输出是否可移植、完整且无私有测试泄露？
