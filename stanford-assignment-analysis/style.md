# Language: Skill Doc + Analysis Output · 语言风格：Skill 文档 + 分析帖输出

Use two voices: austere for the package, analytical and pedagogically plain for
the post. The second inherits the first's evidence discipline, not its academic
distance.
使用两种声音：包内冷峻；帖子分析严谨且教学白话。后者继承前者的证据纪律，不继承其
学术距离。

---

## Voice A — Skill documentation | Voice A——Skill 文档

The package voice is fixed: zero filler; load-bearing verbs; em-dash
qualification; a short sentence to land the stress; honesty over rhetoric.
包内声音固定：零套话；承重动词；破折号用于限定；短句落重音；诚实压过修辞。

1. **Zero filler.** Cut “comprehensive / systematic / worth noting”. State the
   prerequisite, inferential job, artifact, or break.
   **零套话。** 删“全面／系统／值得注意”。写前置、推断任务、工件或断点。
2. **Concrete verbs.** Derive, map, bind, expose, reconcile — not “conduct an
   analysis”. | **具象动词。** 推导、映射、绑定、暴露、对账——不用“进行分析”。
3. **One paragraph, one job.** Cause joins paragraphs; numbering does not create
   causality by itself. | **一段一项任务。** 因果连接段落；编号本身不创造因果。
4. **Honesty over narrative.** A missing artifact is a named gap, not an invitation
   to infer what the student probably produced.
   **诚实压过叙事。** 缺工件就是具名缺口，不是猜学生“可能做了什么”的邀请。

---

## Voice B — Analysis-post output | Voice B——分析帖文体

Write the post **austere in inference, plain in explanation, reflective only
where evidence permits**. Assume a reader lacks course context without treating
them as incapable. Never mention a skill in the deliverable.
帖子须**推断冷峻、解释白话、只在证据允许处反思**。假设读者缺少课程背景，但不把读者当作
无能力者。交付物中不得点名任何 skill。

### Begin with the learning problem | 从学习问题开始

1. **Open on the real capability.** State what this assignment connects that the
   underlying lessons leave separate. Avoid “this assignment comprehensively covers”.
   **以真实能力开篇。** 说明作业把哪些课程中分散的东西连接起来。不写“本作业全面覆盖”。
2. **Give a knowledge map before detail.** The map shows dependency, not merely
   the order in which files or question numbers appeared.
   **细节前给知识地图。** 地图展示依赖，不复述文件或题号出现顺序。

### Make each section independently enterable | 让每节可独立进入

3. **Four functions bracket the evidence.** At entry, name the prerequisite and
   the downstream break without it. At exit, compress the earned takeaway and
   name its next use. Show the source task ID, score/weight, and type nearby.
   Reuse repository labels where they exist.
   **四项功能夹住证据。** 入口点前置，并写缺它下游哪里断；出口压缩经证据赢得的结论，
   再点名下一用途。附近展示来源题号、分值/权重与类型。仓库已有标签时沿用。
4. **Motivation precedes technical objects.** Before the first equation, chart,
   excerpt, or design, the reader must know what problem it is resolving.
   **动机先于技术对象。** 首个公式、图表、片段或设计前，读者先知道它在解决什么问题。
5. **Prompt and analysis remain visible.** Put the source task in one bold-labeled
   block and the reasoning in another; no answer enters the prompt block. Under
   the Stanford reference profile, use `**题目**` and `**解析**` / `**思路与代码**`.
   **题面与分析保持可见。** 来源任务与推理分别置于两个加粗标记块；题面块不放答案。
   Stanford 参照 profile 使用 `**题目**` 与 `**解析**` / `**思路与代码**`。

### Derive; do not announce | 推导，不宣告

6. **One licensed move at a time.** For mathematical work, put one transformation
   per line and name the rule behind non-obvious steps. For empirical work, keep
   setup, observation, interpretation, rival, and limit distinct.
   **每次一个有依据动作。** 数学任务每行一个变换，并点明非显然步骤所用规则；实证任务
   分开设置、观察、解释、竞争解释与局限。
7. **Intuition follows evidence.** Give a concrete analogy after the derivation
   or observation, not instead of it. The analogy must preserve the boundary.
   **证据后再给直觉。** 推导或观察后给具象类比，不用类比替代证据；类比须保留边界。
8. **Name failure precisely.** Not “easy to make mistakes”, but the exact sign,
   normalization, assumption, state, measure, rubric condition, or interpretation
   that fails.
   **精准点名失败。** 不写“容易出错”，写清符号、归一化、假设、状态、测量、rubric 条件或
   解释具体哪里失效。

### Make artifacts readable as arguments | 把工件写成论证

9. **Strategy before excerpt.** Explain the mechanism in ordinary language;
   then show only the code, prose, figure, table, or slide fragment that carries it.
   **片段前先讲策略。** 先用人话讲机制，再展示承载机制的代码、正文、图、表或幻灯片片段。
10. **Every excerpt earns its place.** Follow it immediately with theory ↔ artifact,
   invariant ↔ condition, observation ↔ claim, or rubric ↔ design mapping. No
   orphan excerpts.
   **每个片段须挣得位置。** 紧跟理论↔工件、不变量↔条件、观察↔主张或 rubric↔设计映射。
   不留孤儿片段。
11. **Discuss alternatives through tradeoffs.** Compare what each option exposes,
    hides, costs, or guarantees. “Elegant” and “advanced” are not analysis.
    **用权衡讨论替代。** 比较每个选择暴露、隐藏、付出或保证什么。“优雅”“高级”不是分析。

### Reflect without fiction | 反思，但不虚构

12. **Use first person only when earned.** Ground it in a real revision, supplied
    note, feedback record, or observable design choice. Otherwise use neutral
    analytical narration.
    **第一人称须挣得。** 以真实修订、用户笔记、反馈记录或可观察设计选择为依据；否则使用
    中性分析叙事。
13. **Cross-insight needs comparison.** Use a table when multiple tasks reveal
    the same transformation under different forms; close with one transferable
    method, not a slogan. With fewer than two comparable tasks, mark the section
    not applicable instead of manufacturing a pattern.
    **跨题洞见须经比较。** 多任务以不同形式呈现同一变换时用表格；以一个可迁移方法收束，
    不喊口号。少于两个可比任务时标记不适用，不制造规律。

### Keep it portable | 保持可移植

14. **Match math to the destination.** Under the Stanford reference profile, use
    Unicode/plain text and no LaTeX delimiters for forum portability. Otherwise
    prefer Unicode/plain text when the target
    platform lacks reliable LaTeX; retain repository-native LaTeX when it renders
    correctly. Never mix conventions accidentally.
    **数学形式匹配目标。** Stanford 参照 profile 为论坛可移植性固定使用 Unicode/纯文本，禁止
    LaTeX 定界符。其他课程在目标平台不可靠支持 LaTeX 时优先 Unicode/纯文本；仓库能正确渲染
    时保留其 LaTeX 约定。不得意外混用。
15. **Use ordinary portable references.** Give file/page/section/line addresses;
    avoid editor-only syntax, private grader details, emoji, and production placeholders.
    **使用普通可移植引用。** 给文件/页/节/行地址；不用编辑器专属语法、私有 grader 细节、
    emoji 与制作占位。
16. **Short headings, earned close.** Headings name the concept and action, not
    the grading ID. End by stating what method now transfers elsewhere; add no
    new evidence in the footer.
    **标题短，收束须挣得。** 标题写概念与动作，不塞计分 ID。结尾说明哪个方法可以迁移；
    footer 不添新证据。

## One-pass clean | 一键清洗流程

1. Hoist motivation before every first technical object. |
   把动机提到每节首个技术对象之前。
2. Expand skipped reasoning; state the rule, setup, or evidence that licenses it. |
   补齐跳步，并点明许可它的规则、设置或证据。
3. Delete full-artifact dumps; retain only mechanism-bearing excerpts. |
   删除整工件堆，只留承载机制的片段。
4. Add theory/invariant/observation/rubric mapping after every excerpt. |
   每个片段后补理论/不变量/观察/rubric 映射。
5. Replace invented first-person turns with artifact-supported narration. |
   把虚构第一人称转折改为工件支持的叙事。
6. Normalize math and references to the target platform; remove private details. |
   按目标平台统一数学与引用形式；删除私有细节。
7. Randomly read one section alone: it must explain what, why, how, evidence,
   and next. | 随机单读一节：须说明做什么、为何、怎么做、证据与下一步。
8. Check every section exposes its source unit and keeps prompt separate from
   analysis. | 检查每节均显露来源单元，并把题面与分析分开。
